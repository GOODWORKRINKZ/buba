## 1. Ключевые выводы

1. **4 модуля BU04:** 3 якоря на роботе (равносторонний треугольник) + 1 тег на человеке
2. **Режим TWR (двусторонняя дальнометрия):** DS-TWR точнее PDOA для 360° следования
3. **Сопроцессор RP2040:** PIO даёт неограниченное число UART для подключения якорей
4. **Аналитическая формула позиционирования:** Патент CN105828431A — замкнутое решение без итераций
5. **Data-over-UWB для кнопки:** Передача команд через тот же радиоканал, без доп. железа
6. **Точность:** ±10 см (патент) до ±15 см (сообщество) после поканальной калибровки

## 2. Аппаратная архитектура

```
                   ┌────────────────────────────────────┐
                   │     RP2040 (UWB Co-Processor)       │
                   │                                    │
  BU04-A──UART0──→ │  d1 ┐                              │
  BU04-B──UART1──→ │  d2 ├─ формула (L,θ) ──UART────→  │ ROBOT
  BU04-C──PIO ───→ │  d3 ┘  + фильтр        I2C ───→   │ (ROS2/STM32)
                   │                                    │
                   │  Приём "BTN:1" из UWB data frame   │
                   └────────────────────────────────────┘

  ТЕГ (на человеке):
    BU04 + LiPo 500мАч + внешняя всенаправленная антенна (IPEX)
    TWR-ответы: автоматические (стоковая прошивка)
    Кнопка: GPIO → dwt_writetxdata("BTN1") → UWB радиоканал
```

### Почему RP2040 а не ESP32

| Параметр | RP2040 | ESP32 |
|----------|--------|-------|
| PIO (неограниченные UART) | ✅ Да | ❌ Макс. 3 аппаратных |
| Детерминированность | ✅ Без ОС | ❌ FreeRTOS джиттер |
| Потребление | ~20 мА | ~80+ мА |
| Цена | ~$4 | ~$5 |

### Почему UART для BU04↔RP2040

Стоковая прошивка BU04 — ТОЛЬКО UART (AT-команды). Аппаратный I2C есть на пинах PB6/PB7, но прошивка на него не отвечает. Кастомная прошивка STM32 через SWD — огромные трудозатраты. RP2040 PIO даёт неограниченные UART бесплатно.

### Протокол выхода на робота

```
TAG:150,25\r\n      — тег на 150 см, +25° азимут
TAG:120,0\r\n       — тег на 120 см, прямо по курсу
TAG:LOST\r\n         — все 3 якоря потеряли тег
TAG:NOISE\r\n        — замеры слишком зашумлены, держать позицию
```

ASCII, 115200 бод. Один `scanf("%*3s%f,%f")` на стороне робота. 15 байт на пакет при 5 Гц = 75 байт/с — 0.06% от пропускной способности.

## 3. Формула позиционирования (Патент CN105828431A)

**Университет Шанхая по науке и технологии (2016)** — основной референс.

### Схема

```
3 базовые станции A, B, C в равностороннем треугольнике (сторона = a)
Тег M — неизвестная позиция
Измеренные расстояния: d1, d2, d3

                  C
                 / \
                /   \
               /  M  \
              /   •   \
             /         \
            A───────────B
            ←──  a  ──→
```

### Формула

```
L = √((d1⁴ + d2⁴ + d3⁴ − d1²d2² − d1²d3² − d2²d3²) / (3·a²))

θ = arctan(√3(d3² − d2²) / (−2d1² + d2² + d3²))
```

**Выход:** (L, θ) — полярные координаты тега от центра робота
**Заявленная точность:** ±10 см
**Преимущества:** Без итераций, без накопления ошибки, каждое измерение независимо

### Система координат

- O(0,0,0) = середина стороны AB
- Ось y = направление вперёд робота (⊥ BC, через H)
- H = центр равностороннего треугольника
- M'(x,y) = проекция тега на плоскость базовых станций

## 4. UWB модули Ai-Thinker: сравнение BU03 vs BU04

### Таблица AT-команд (29 команд)

*(полная таблица в *.planning/research/ROBOT-FOLLOWING.md §14.7)*

**Ключевые команды для TWR-трилатерации:**

| Команда | Назначение | BU03 | BU04 |
|---------|-----------|------|------|
| AT+SETUWBMODE=2 | Установить TWR режим | ✅ | ✅ |
| AT+DISTANCE | Получить TWR расстояние | ✅ | ✅ |
| AT+ADDTAG | Добавить тег в список якоря | ✅ | ✅ |
| AT+RNGOFF | Калибровка смещения дальности | ✅ | ✅ |
| AT+USER_CMD | Пользовательская команда (точка расширения) | ✅ | ✅ |
| AT+SAVE | Сохранить конфигурацию | ✅ | ✅ |

**Вердикт:**
- **BU03 как ТЕГ:** ✅ ДА. Всенаправленная антенна (IPEX) — лучшее 360° покрытие
- **BU04 как ЯКОРЬ:** ✅ ДА. Направленный луч 120° × 3 в треугольнике = 360°

### Анализ SDK (cmd_fn.c)

- ОДИНАКОВАЯ прошивка для BU03 и BU04 — нет `#ifdef BU03`/`#ifdef BU04`
- `AT+USER_CMD` (f_user_cmd) — пустая функция-заглушка, сюда добавляем обработчик кнопки
- `dwt_writetxdata()` доступна через HAL DW3000 — отправка своих данных по UWB
- Добавление своей AT-команды: 1 строка в массив `known_commands[]`

**Кнопка "следуй за мной" — реализация (~20 строк C):**
```c
// В cmd_fn.c:
int f_button(int opt, int argc, char* argv[]) {
    // Читаем GPIO пина кнопки
    // Отправляем через UWB: dwt_writetxdata("BTN:1\r\n", 7);
    return 0;
}
// В known_commands[]:
{"AT+BUTTON", f_button},
```

## 5. Процедура калибровки TWR

**Источник:** Qorvo APS011 "Источники ошибок в схемах TWR" (22 стр) + APS014 "Калибровка задержки антенны"

### Источники ошибок TWR

| Источник | Величина ошибки | Компенсация |
|----------|----------------|-------------|
| Дрейф часов (кварц 20ppm) | DS-TWR: ±2-3 см, SS-TWR: ±20-60 см | Использовать ТОЛЬКО DS-TWR |
| Задержка антенны | 515 нс (≈77 м эквивалент, остаток 0.5-1 нс ≈ 15-30 см) | Поканальная калибровка |
| Многолучевость | ±5-50 см (зависит от помещения) | Leading edge detection, длинная преамбула |
| Шум (SNR) | ±1-5 см | Усреднение 10+ замеров |

### Пошаговая процедура (для каждого якоря A, B, C):

1. Разместить тег на расстоянии **РОВНО 2.000 м** от якоря (лазерный измеритель)
2. Установить TWR режим: `AT+SETUWBMODE=2`
3. Собрать **100 замеров**: `AT+DISTANCE`
4. Вычислить среднюю ошибку: `смещение_см = среднее(измерения) − 200`
5. Применить коррекцию: `AT+RNGOFF=<round(смещение_см)>`
6. Сохранить: `AT+SAVE`
7. Проверить: остаточная ошибка должна быть **< ±5 см**

**Калибровочное расстояние:** 2.0 м для каналов 5 (6.5 ГГц) и 9 (8 ГГц) — APS011 Таблица 3

### 6.1. Анализ GitHub проектов

#### yws94/UWB_Multi-Ranging_DW3000 — КЛЮЧЕВОЙ ПРОЕКТ ⭐⭐⭐⭐⭐
**URL:** github.com/yws94/UWB_Multi-Ranging_DW3000

**Что внутри:** Реализация одновременного ranging с несколькими якорями на DW3000. В отличие от стандартного подхода (последовательный опрос якорей), этот проект реализует параллельный ranging — все якоря меряют расстояние одновременно.

**Почему это важно для нас:** Наша архитектура с 3 якорями на роботе требует быстрого получения d1,d2,d3. Последовательный опрос (AT+DISTANCE с якоря A, потом B, потом C) даёт задержку 300-600 мс. Параллельный ranging сокращает это до ~100 мс — рост с 1.6-3.3 Гц до 10 Гц.

**Что конкретно берём:** Алгоритм scheduling запросов — как три якоря не мешают друг другу. Формат UWB фреймов для multi-ranging. Код на C++ (ESP32), портируем на STM32 внутри BU04.

**Ограничения:** Работает через прямой SPI к DW3000 (не AT-команды). Для BU04 потребуется кастомная прошивка STM32. Не проверялось на BU04.

#### HeadTriXz/DWM3000-ESP32 — Альтернативная библиотека DW3000 ⭐⭐⭐⭐
**URL:** github.com/HeadTriXz/DWM3000-ESP32

**Что внутри:** Arduino-библиотека для DW3000 — независимая от Makerfabs/NConcepts. Чище код, лучше документация, активная разработка.

**Сравнение с Makerfabs:** У Makerfabs библиотека от NConcepts (3 года без обновлений). HeadTriXz — свежая (2024+), с исправленными багами SPI-таймингов. Лучше подходит для production.

**Что берём:** Если будем писать кастомную прошивку BU04 (прямой SPI к DW3000) — брать эту библиотеку как основу, а не Makerfabs.

#### MISTLab/RangeTracker — Single-anchor tracking ⭐⭐⭐⭐
**URL:** github.com/MISTLab/RangeTracker

**Что внутри:** Точное отслеживание позиции с ОДНИМ UWB якорем. Использует не трилатерацию, а анализ изменения расстояния во времени + робот движется (synthetic aperture).

**Прорыв для нас:** Если один якорь может давать позицию — нам не нужно 3 якоря на роботе! Но есть нюанс: робот должен двигаться чтобы получить угловое разрешение. Для статичного робота не работает.

**Вердикт:** Не подходит для v1 (нужна статическая локализация), но ОЧЕНЬ интересно для v2 — потенциально сокращает BOM с 4 BU04 до 2.

#### advoard/advoard_localization — ROS + UWB ⭐⭐⭐⭐
**URL:** github.com/advoard/advoard_localization

**Что внутри:** Готовый ROS-пакет для локализации с UWB. Интегрируется с navigation stack (move_base, AMCL). Написан на C++, catkin-совместим.

**Что берём:** Когда дойдём до ROS2-интеграции (фаза 6) — использовать этот пакет как референс. Формат сообщений, топики, трансформации.

#### jremington/UWB-Indoor-Localization_Arduino ⭐⭐⭐
**URL:** github.com/jremington/UWB-Indoor-Localization_Arduino

**Что внутри:** Open source indoor localization на Arduino + ESP32 UWB модулях. Простой код, хорошо задокументирован.

**Что берём:** Минимальную реализацию трилатерации на микроконтроллере — можно портировать на RP2040. Код трилатерации (~50 строк C++).

### 6.2. Анализ научных статей

#### Single UWB anchor position tracking (arXiv:2005.10648) — ПРОРЫВ ⭐⭐⭐⭐⭐
**Авторы:** Cao et al., 2020. **Цитирований:** 50+

**Что сделали:** Доказали что ОДИН UWB якорь может давать точную 2D-позицию тега. Метод: робот движется → изменение расстояния во времени + одометрия → оценка угла через synthetic aperture. Точность: 21 см (один якорь) против 14 см (три якоря).

**Влияние на наш проект:** Радикально! Если один BU04 может давать позицию — зачем 3 якоря? НО: метод требует ДВИЖЕНИЯ робота. Для статичного робота не работает. v1 делаем на 3 якорях, v2 экспериментируем с single-anchor.

**Что конкретно берём:** Алгоритм synthetic aperture — математика оценки угла по последовательным замерам расстояния. Код на Python в открытом доступе (MISTLab/RangeTracker).

#### UWB Radar on Mobile Robots (arXiv:2512.01018, 2025) — СВЕЖАК ⭐⭐⭐⭐
**Что сделали:** Использовали DW3000 НЕ для ranging, а как РАДАР — детекция препятствий по отражённому сигналу. Тот же чип, другая функция.

**Прорыв:** Один DW3000 может одновременно измерять расстояние до тега И детектировать препятствия. Двойное использование железа без дополнительных сенсоров.

**Вердикт:** v2+ — добавить радарный режим для obstacle avoidance. Сейчас не приоритет.

#### Fusing Odometry, UWB, Spatial Detections (arXiv:2304.06264) ⭐⭐⭐⭐
**Что сделали:** Слияние одометрии колёс + UWB ranging + пространственные детекции (камера/LiDAR) для относительной локализации нескольких роботов.

**Что берём:** Подход к multi-sensor fusion. Конкретно: как объединять UWB (абсолютная позиция) с одометрией (относительное перемещение) через EKF. Нам это нужно в фазе 5 для робота.

#### UWB-IMU Sensor Fusion (IEEE, 2023) ⭐⭐⭐⭐
**Что сделали:** Экспериментальная оценка fusion UWB+IMU. Сравнили EKF, UKF, particle filter. Результат: EKF достаточно для ±10 см, UKF лучше при нелинейностях, particle filter избыточен.

**Что конкретно берём:** Подтверждение что EKF достаточен для нашей точности. Не нужен сложный UKF. Параметры EKF: process noise, measurement noise — готовые цифры из статьи.

#### Optimal 4-anchor layout (ScienceDirect, 2024) ⭐⭐⭐
**Что сделали:** Математически вывели оптимальное размещение 4 якорей для максимальной точности. Результат: якоря в вершинах квадрата лучше чем в линию.

**Применимость:** У нас 3 якоря (треугольник). Статья подтверждает что равносторонняя геометрия близка к оптимальной. При переходе на 4 якоря (v2 ETH-подход) — использовать квадрат.

#### SALOS Single-Anchor Localization (PubMed, 2024) ⭐⭐⭐
**Что сделали:** UWB локализация с одним якорем на основе 3D-лучей. Не требует движения (в отличие от arXiv:2005.10648). Основан на анализе многолучевости (multipath).

**Вердикт:** Экспериментальная технология, требует сложной обработки сигнала. Не для v1.

## 7. Патентный ландшафт — анализ

### 7.1. Ключевые патенты

#### CN105828431A (2016) — НАШ РЕФЕРЕНС ⭐⭐⭐⭐⭐
**Университет Шанхая по науке и технологии**

**Суть:** Замкнутое аналитическое решение для 3 базовых станций в равностороннем треугольнике. Формулы L и θ через d1,d2,d3. Без итераций, без Калмана.

**Анализ применимости:**
- **Сильные стороны:** Простота вычислений (RP2040 справится). Каждое измерение независимо — нет накопления ошибки. Не требует настройки фильтров. Работает внутри и снаружи.
- **Слабые стороны:** Требует 3 расстояния — потеря 1 якоря = потеря позиции. Не учитывает высоту тега (z) — работает в 2D. Формула чувствительна к шуму при малом a (a² в знаменателе).
- **Вердикт:** Идеально для v1. Достаточно для ±10-15 см. Для v2 можно улучшить Калманом.
- **Статус:** Действует в Китае. Не действует в РФ/ЕС/US — можно использовать свободно.

#### CN113282085A (2021) — UWB + зрение ⭐⭐⭐⭐
**Суть:** Система следования с fusion UWB и видеоданных. UWB даёт грубую позицию, зрение уточняет и идентифицирует цель.

**Анализ:** Пересекается с нашей архитектурой v2. Показывает что UWB+зрение — стандартный путь развития после чисто UWB v1.

#### CN116300613A (2023) — UWB + IMU + поведенческий контроль ⭐⭐⭐⭐
**Суть:** Мульти-сенсорный фьюжн (UWB+IMU) с behaviour-aware контроллером. Самый современный китайский патент по теме.

**Анализ:** IMU помогает когда UWB временно теряет тег (inertial dead reckoning). Behaviour-aware — робот предсказывает движение человека. Полезно для v3.

#### US20230008482A1 (2023) — US патент multi-anchor following ⭐⭐⭐
**Суть:** Американская заявка на multi-anchor UWB следование. Покрывает конфигурации с 2+ якорями на роботе.

**Анализ:** Широкий патент. Может блокировать коммерческое использование в US. Для личного/учебного проекта не проблема.

#### KR20230007877A (2023) — Корейский патент ⭐⭐⭐
**Суть:** Корейский аналог. Возможно, от Samsung или LG.

**Анализ:** Подтверждает интерес крупных корпораций к UWB следованию.

### 7.2. Тренды патентного ландшафта

```
2016: CN105828431A — чистая UWB трилатерация (наш подход)
2021: UWB + зрение (fusion) 
2022: UWB + объезд препятствий (практическое применение)
2023: UWB + IMU + поведенческий контроль (интеллектуализация)
2025: 耀晟智能 — коммерческий продукт (индустриализация)
```

**Вывод:** Чисто UWB следование (2016) — решённая задача. Фронтир (2021-2025) — сенсорный фьюжн и поведенческий ИИ. Наш v1 на чистом UWB — правильный старт. v2+ — добавлять зрение/IMU.
