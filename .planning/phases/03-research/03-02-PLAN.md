    Specific guidance per gray area:

    G1 (Triangle side a):
    - Research says: patent formula works for any a, but larger a = better geometry (distance differences more distinct), smaller a = fits smaller robot chassis
    - RECOMMEND: Start with a=40cm as compromise. The patent formula has a² in denominator — larger a reduces noise amplification. 40cm fits a ~50cm robot chassis (RPi robot, TurtleBot, custom). If robot is smaller, drop to 30cm with expected ~20% accuracy reduction.
    - OPEN: Final a determined by robot chassis dimensions in Phase 5. Phase 4 calibration should test TWR accuracy at 30cm and 50cm baseline to quantify the accuracy-vs-size tradeoff.

    G2 (RP2040 poll rate):
    - Research says: BU04 TWR cycle time is ~100-200ms per range (3 anchors = 300-600ms total if sequential). BUT anchors can range in parallel if tag responds to all — need to confirm BU04 stock firmware behavior.
    - RECOMMEND: Target 5Hz (200ms cycle). Poll each anchor's UART buffer in round-robin at 50ms intervals. Collect d1,d2,d3 within a 200ms window, solve once per window. If TWR cycles overlap (anchor A gets response while B is ranging), effective rate could reach 10Hz.
    - OPEN: Actual TWR cycle time must be measured empirically in Phase 4. BU04 AT command latency unknown. Plan for 5Hz, hope for 10Hz.

    G3 (Output protocol):
    - Research says: ASCII "TAG:L,θ" format already designed in RESEARCH.md section 2 with LOST and NOISE states.
    - RECOMMEND: Use ASCII format as designed. Rationale: (a) human-debuggable with any serial terminal, (b) single scanf("%*3s%f,%f") parse on robot side, (c) 15 bytes per packet at 5Hz = 75 bytes/sec — negligible at 115200 baud (0.06% utilization), (d) binary saves 11 bytes/packet — not worth losing debuggability for 0.05% bandwidth.
    - OPEN: None. ASCII format locked unless robot MCU has extreme RAM constraints (<1KB).

    G4 (2-of-3 anchors lost):
    - Research says: With 2 distances, position is ambiguous (two possible intersection points of circles). The patent formula requires all 3 distances. ETH Zurich paper handles occlusion via IEKF with 4 receivers — but we have 3.
    - RECOMMEND: If 2-of-3 anchors lost → output TAG:LOST and stop robot. Do NOT attempt 2-anchor estimation — the ±y ambiguity means robot could turn wrong direction. 2-anchor fallback adds code complexity for a degraded mode that could cause unsafe motion. Better: stop and wait. If one anchor is consistently dropping (antenna orientation, obstruction), that's a mounting/placement fix, not a software fix.
    - OPEN: If anchor dropout is frequent in practice (Phase 5 testing), consider adding a 4th anchor (ETH Zurich approach) or using last-known-good position with timeout.

    G5 (TWR per-module calibration):
    - Research says: APS011 documents TWR error sources: antenna delay (dominant), clock drift, signal path. APS014 describes antenna delay calibration procedure using known-distance measurement. Each BU04 module may have different antenna delay due to manufacturing variance.
    - RECOMMEND: YES — per-module calibration required for ±10cm accuracy. Procedure: place tag at exactly 2.000m from each anchor (measured with laser/tape), collect 100 TWR ranges per anchor, compute mean offset. Apply AT+RNGOFF per anchor. Do NOT assume BU04 modules are identical — antenna delay can vary 0.5-1.0ns (~15-30cm error). This is Phase 4 work.
    - OPEN: Whether AT+RNGOFF command exists in BU04 stock firmware (BU03/BU04 AT command doc references RNGOFF — verify empirically in Phase 4). If not, offset must be applied on RP2040 side.

    G6 (Tag battery + sleep):
    - Research says: BU04 datasheet power in TAG mode: ~50mA continuous (TWR responding). 500mAh LiPo = ~10h runtime. No sleep mode in stock firmware.
    - RECOMMEND: v1: no battery indicator, no auto-sleep. 10h is sufficient for testing sessions. Add physical power switch on tag. v2 (Phase 6+): add voltage divider to BU04 ADC pin + AT command to query, RP2040 relays voltage to robot; auto-sleep requires custom STM32 firmware (out of scope for Phase 3-5).
    - OPEN: Whether BU04 exposes battery voltage via any AT command (check AT command doc). If ADC pin is accessible on BU04 header, external voltage divider → RP2040 ADC is alternative.

    G7 (Kalman location):
    - Research says: D9 already decided — patent analytic formula, no Kalman for initial version. IEEE papers (ETH, IFAC) use Kalman/IEKF for smoothing.
    - RECOMMEND: No Kalman on RP2040 in v1 (per D9). If measurements are noisy in Phase 5 testing (jitter >±20cm), add a simple 1D Kalman filter on L and θ independently on the ROBOT side (not RP2040). Rationale: robot MCU (STM32/ROS2) has more compute headroom than RP2040; RP2040 stays simple/deterministic. Start with moving average (N=3) on RP2040 as lightweight smoothing.
    - OPEN: Noise level unknown until Phase 5 grid testing. Decision deferred to Phase 5 based on empirical noise data.
